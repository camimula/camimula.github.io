SQLite Core, Imperative Shell

===== 概要 =====
・ソフトウェア開発において価値があるのは実装よりもテスト
	・惰性の単体テストをやめよう
	・可能な限りE2Eに近いテストの自動化を検討すべき
・SQLite Core, Imperative Shell
	・APIレスポンスやサーバー型DBの必要なデータをSQLiteにスナップショットし、アプリケーションはSQLiteを参照する
	・単体テストと結合テストの境界が非常に薄くなる
	・プロパティベーステストがE2E寄りで現実的になる

===== 0. 議論の射程 =====
以下の議論は、世の大部分を占めるであろう、ビジネスのためのソフトウェアを念頭においている。ソフトウェアのためのソフトウェアについても基本的な発想は大きく変わらないと思うが、整合が取れない部分についてはご容赦願いたい。

さらには、UIを持つソフトウェア...いわゆるフロントエンドに対しても、射程圏外だろうと認識している。つまり、バックエンドと呼ばれるものを対象とする。かなり限定的には見えるが、htmxのようなバックエンドに多くを寄せる動きがある現在、それなりに関心の大きな部分でもあることだろう。

===== 1. テストの価値 =====
ソフトウェアは目的というよりは方法だろう。その意味で、目的を実現したのか問われるのは当然だが、それに解答できるのはテストのみである。いかに実現したか(アーキテクチャ・コード)は何を実現したのかを保証しない点に留意が必要だろう。従って、まず最初に検討されるべきはテストだと私は思っている。

要はTDDである。しかし、TDDは実装レベルの方法論に歪められてしまった節を感じる。重要なのは目的の実現が達成されているかを問うことで、その上で余計な文脈の混じらない直接的な方法を尊ぶ、つまり書くべきテストとはE2Eなのだ。E2Eは実現されるべきことを直接表現することができる、おまけに理屈の上では要件の変更以外で壊れることはないので実装よりも長生きするはずだ。

しかし、残念なことにE2Eにはコスト、制約、脆さ...等々厳しい壁がある。もう少し小さく、結合テスト、単体テストと妥協が続いたのが、テストピラミッドの文脈だろう。それが最終的には単体テストを厚くすれば良いという教義として受容されてしまったのではないかと睨んでいる。

教義と否定的なのは、単体テストは実装のテストに堕ちやすい点だ。先にも述べたが実装は目的の達成を保証しない、単体テストの価値については常に懐疑的であり続ける必要がある。単体テストを書く上では、Functional Core, Imperative Shellで効果を最大化する等々の戦略が伴うべきであろう。

結局のところE2Eテストの妥協という視点を持ち続けるべきなのだ。その妥協は妥当だが、どこまで妥協すべきかについては常に検討し続けねばならない。

脱線するが、テストの効果に対してあまり検討されていない余談を加えよう。テストは何を実現すべきかの指示であり、実現された内容の保証である。これを契約として、各々の裁量に任せるのが一つ理想的な開発スタイルだと私は信じている。詳細設計の自由を各々の裁量に任せるのが開発におけるインセンティブの提供の一形態だと感じているのだ、これこそ「包括的」な開発と言えるのではないだろうか。アセモグルの理論とのアナロジーを楽しむのも良いが本題ではないので割愛しよう。

そして、レビュー負担が叫ばれている生成AIコーディングに対する処方箋にもなることだろう。非機能面も、セキュリティスキャンや負荷試験で大抵の現場は妥協ないし、十分に保証したと言えるだろう。割り切りにもよるが、レビューというプロセスの負担を大幅、場合によっては完全に取り除くことも決して不可能ではないはずだ。

===== 2. SQLite Core, Imperative Shell =====
先ほどE2Eの妥協と述べたが、少々訂正しよう。仮にE2Eテストの問題を解決できたとしても、ほとんど超えようのない壁が一つある。ネットワークという物理的制約に由来する実行速度だ。解消策は並列実行になると思うが、E2Eテストの網羅性を高めるために、膨大な事前状態をプロパティベースで再現しようとした時、スケールさせるのはあまり現実的ではないだろう。

そこで、発想を逆転させて、高速実行可能な単体テストをどこまでE2Eに近づけられるかの検討を行うのも手だろう。これが私の提唱する「SQLite Core, Imperative Shell」である。至って簡単な主張で、下記のようにアプリケーションとサーバー型RDBの間にSQLiteを挟むだけである。

サーバー型RDB → SQLite ←→ アプリケーション

サーバー型RDBの必要なテーブル・必要な行をSQLiteにスナップショットし、アプリケーションはSQLiteに対してクエリを発行する。スナップショットする際、サーバー型RDBとSQLiteの間は簡単なWHERE句だけを認めるようにすれば、実環境との挙動差はほとんどなくなる。つまり、単体テストを結合テストのレベルに近づけることができる。

この方針の利点は、テスト効果を増進させるだけではない。アプリケーション内でフィルターや集約がSQLで記述できるようになるのも強烈な効果だろう。SQLはリレーショナル完備であるが故に、大抵の用途は十分に捌ける。コードの役割は主に接着剤となる、制御文やSQLのサポートしていない値の加工などかなり限定的なものに収まることだろう。また、下手にコードを書くよりもSQLiteの長年の最適化に委ねた方がパフォーマンス面が良いケースが散見されることだろう。

先述のように数学的背景と長年の採用実績があるため、SQLは非常に安定したデータアクセスインターフェースである。SQLiteに至っては2050年までのサポートが保証されているので、ロジックの核が非常に強固になることが保証される。周辺環境の進展が早くとも、核が保証され続けるのであれば、更新に追従するのも相当楽になることだろう。

===== 3. 見えている限界 =====
銀の弾丸は存在しない、この方法にも限界はある。現時点で見えている範囲について正直な意見を述べておこう。

3.1 RDBを前提にしている問題
意図してサーバー型RDBと書き続けていたことについて述べよう。世のデータソースはRDBだけではない、NoSQLもあれば外部API(JSON)もあり得る。それらのデータをSQLiteに入れるのかという問いは正当だろう。

正直なところ、外部APIに大きく依存する構成であればほとんど無力だと認めよう。そのような構成の場合は、本質的にはモックの問題になることだろう。ロジックを持つのであれば、その検証はFunctional Core, Imperative Shellを採用してテストした方が余程筋が良い。逆にいえばそれだけの話ではある、昇格できないなら妥協点をどこに持ち込むかという当初の話に過ぎない。

データソースの多様性に関しては、余程特殊な非機能要件がない限りRDBで事足りると主張したい。今やNewSQLが存在しており書き込みスケール問題はほぼ解決している。そもそも、そこまでスケールするサービス自体ほとんど存在しないし、管理コストの追加や、トランザクションサポートを捨ててまで他のDBを採用する理由について見出す場面というのもかなり限定的だろう。

データの構造の多様性で言えば、問題自体が存在しないと認識している。そもそもSQLiteはJSONに対応しているので致命的になることは考えにくい。その上、スキーマレスの現実はスキーマの遅延評価に過ぎないことに留意すべきだろう。本当にスキーマが事前に定義できないものは、プログラムで利用する事自体困難であり、相当特殊なデータなので個別の対応が必要なのはやむを得まい。

3.2 書き込みの問題
先の方法論では意図して書き込みに触れていない。Arrange・Actまでは議論しているが、Assertの対象については触れていないのも認めざるを得ない。

例えば、Assertの対象がAPIサーバーのレスポンス(JSON等)である分には何も困らないが、DBのデータの状態の場合は問題が発生する、SQLiteのデータを見るのか、サーバー型RDBのデータを見るべきなのか。

当然、正しいのはサーバー型RDBを確認することである。INSERTやUPDATEの挙動に関してはどのDBも大差ないという割り切りの下、SQLite内に書き込み対象テーブルを置いて、そこに反映されたデータをサーバー型RDBに反映する方針にするのが一つの落とし所だとは思う。

それでも、サーバー型RDBのINSERT/UPDATE挙動と合致するかの保証や、競合に関しては不十分になるだろう。単体テストとしての価値がどうしても薄れるのは認めざるを得ないし、そういう意味で結合テストレベルに到達したというのも不誠実な主張であった。これが最大の構造的欠陥だろう。とはいえ、書き込みだけ実DBを用いるなり、割り切るなりしても十二分に効果的だ。

3.3 依然E2Eに到達しない
E2Eを尊ぶと主張したが、提案した方法では単体テストの昇格は限定的である。サーバー型RDBをSQLiteで置き換えることで高速化を狙う方針もあると思うが、それは単なるモックだ。テスト価値を下げる上に方言の吸収や挙動差が問題になる。この辺がやむを得ない妥協点になることだろう。最終的にE2Eテストは必須である。とはいえ、結合テストレベルは比較的要件を表現する能力が高いため、それが分厚ければ十分に効果的なのもまた事実だろう。

3.4 テストエコシステムの未成熟
網羅性のためのプロパティベーステストと唱えた。関数に対してのプロパティベーステストのエコシステムは成熟しているが、SQLに対しては未成熟というのも事実だろう。これに関しては、残念ながらエコシステムの成熟を待つか、自ら作成することが要求されることになるだろう。

プロパティベース基盤の作成について気楽な雑談を展開しよう。結合テストレベルでプロパティベーステストをするにしても、データを作成したい単位は、個々のテーブルというよりはもう少し大きな単位(会員等)になるだろう。複数テーブルに跨って整合性の取れたデータ作成する上では、Factoryパターンの発想は十分に候補になるだろう。その道で既に目覚ましい成果を挙げているFactoryBotを参考にするのは有意義ではないかと考えている。

思うに、DDDを代表に普及したドメインの理解とコードへの対応という概念は、オブジェクト指向の成果だろう。しかし、その方法論は複雑怪奇に膨らみ、開発の負担になっている点は否定し難いと思う。結局概念と機能を対応づけることに関しては距離があるのだろう、ただデータに関してはかなり近いはずだ。テストの文脈でこそ成長すべきものだったのかもしれないと最近は妄想している。

===== 4. まとめ =====
第1節でE2Eに近いテストの重要性について語った。第2節でSQLiteを用いた単体テストをE2Eに近づける工夫について語り、第3節でその限界についても指摘した。

「SQLite Core, Imperative Shell」という方法論を提唱した。一考の余地がある面白いものだと思っている、車輪の再発明でないことを祈るばかりだ。名前からもわかるように、Functional Core, Imperative ShellとSQLiteが着想の大きな母体である。それぞれの生みの親であるGary氏、Hipp氏には感謝したい。

とはいえ、個人的には方法論を推したいというよりは、むしろテストの価値に対する問いを広げたい。この記事を書き始めた理由であるが、最近「テンプレート生成はAIの得意な領域なのでテストコードも書かせるべきだ」と主張する意見を見かけたのが発端だ。AIに書かせるかの判断は自由だが、テストは今までの議論のように、テンプレート埋めのような退屈な作業ではあり得ないと思っている。

そのため、まずはテストをテンプレートとして捉える原因であろう単体テストについて、惰性で生成しても価値がないという意見を改めて繰り返したい。明に暗に示せたと思うが、テストはその重要性にも関わらず相当未開拓な鉱床で、かなりの創造的な仕事の可能性が眠っていると再度主張したい。

諸賢にもこれを機にテストの可能性について今一度探求して頂ければ幸いである。
